## 🐳 Part 1: 数据库环境全面 Docker 化

### 1.1 变更核心
为了解决每台电脑本地 MySQL 配置不一致的问题，我们现在统一切换到 **Docker 容器数据库**。
*   **统一端口**: `3307` (映射到本机，即使你本地装了 3306 也不冲突)
*   **统一密码**: `hy_root_password`

### 1.2 你的操作步骤 (仅需两步)
**步骤 1: 启动数据库**
在项目根目录下运行：
```bash
docker-compose up -d hy-db
```

**步骤 2: 启动后端**
直接运行代码，无需修改任何配置（后端已默认适配 Docker 环境）：
```bash
cd hy-backend
.\mvnw spring-boot:run
```

### 1.3 如果你是数据库管理员
修改表结构后，请：
1.  连上 `localhost:3307` (密码: `hy_root_password`) 进行修改。
2.  导出 SQL 脚本覆盖根目录下的 `_localhost-xxxx-dump.sql`。
3.  提交 Git。

## 📏 Part 2: 规格校验逻辑重构 (Critical Update)

我们重写了 `SpecValidator.java` 工具类，修复了旧代码中严重的逻辑漏洞。

### 2.1 修复了什么？
*   **旧版本的问题**: 不区分单位。如果鱼要求 `≥500g`（重量），系统会把 `20cm`（长度）错误地按数值 20 < 500 的逻辑判为“不达标（违规）”，这虽然结果看起来是对的，但逻辑是错的。
*   **新版本的改进**: 引入了物理维度检查。
    *   **Length (长度)**: `cm`, `mm`, `m`, `厘米`, `米`
    *   **Weight (重量)**: `kg`, `g`, `斤`, `公斤`

### 2.2 现在的行为
当用户填写的单位与规则要求的单位**维度不一致**时（例如：规则是重量限制，用户填了长度），系统将：
*   ❌ **即使数值满足大小关系，也会判定为校验失败。**
*   🚨 **返回明确提示**: `"单位类型不匹配（规则要求:kg，实际填写:cm），请检查填写单位"`

### 2.3 开发须知
*   请在测试时注意新的校验提示。
*   如果前端页面收到 "单位类型不匹配" 的错误，请检查是否是测试数据填错了单位。

## ❓ 常见问题 Q&A

**Q: 我拉代码后报错 `Table 'xxx' doesn't exist` 怎么办？**
A: 说明队友更新了数据库结构。请执行：
```bash
docker-compose down -v  # 删除旧容器数据
docker-compose up -d    # 重新加载新 SQL
```

**Q: 我可以用自己的本地 MySQL 吗？**
A: 可以，但不推荐。请复制 `.vscode/launch.json.example` 为 `launch.json` 并填入你的本地配置。后端会优先读取此配置文件。
